## Ports Ports Ports
We're no longer [on a boat](/blog/on-a-boat), but let's talk about ports.

When you host a TCP server, you open a port (ex. 443 for HTTPS) and listen for incoming connections.
The TCP client will randomly select an ephemeral port to use, and the connection is identified by the source/destination IP/ports.
For example, a connection might be identified as `123.45.67.89:54321 -> 192.168.1.2:443`.

Btw this is usually called a 5-tuple (the 5th item is the protocol), but I think that name is lame.
IP/port it is.

But there's a major problem with TCP... client addresses can change.
When your phone switches from WiFi to cellular, oops your IP changes.
NATs can also arbitrarily change your source IP/port because of course they can.

Whenever this happens, bye bye TCP connection, it's time to dial a new one.
And that means an expensive TCP + TLS handshake, which takes at least 2-3 RTTs.
The users notice the network hiccup.

## WebRTC As Designed
WebRTC tried to solve this issue but it made things worse.
Seriously.

A WebRTC server is *supposed* to allocate an ephemeral port for each connection.
That's because a WebRTC session is actually identified by the destination IP/port; the source is irrelevant.
If the user's source IP/port changes, oh hey that's still Bob because the destination port is the same.

As OpenAI describes, this causes issues at scale because...
- We only have a limited number of ephemeral ports available (10,000-65,535)
- Firewalls love to block ephemeral ports.
- Kubernetes lul

You could probably abuse IPv6 to work around this, but IDK I never tried.
Twitch didn't even support IPv6...

## WebRTC In Practice
So most services end up using a single port instead.

At Twitch I literally hosted my WebRTC server on `UDP:443`.
That's *supposed* to be the HTTPS/QUIC port, but lying means we can get past more firewalls.

Discord actually uses 50000-50032 so each CPU core can get its own port.
As a result it gets blocked on more corporate networks, but you should be doing work anyway.

**HOWEVER, HUGE PROBLEM**.

WebRTC is actually a bunch of protocols in a trenchcoat, and 5 of of those go over UDP directly.
It's [not hard](https://datatracker.ietf.org/doc/html/rfc7983) to figure out which protocol a packet is using, but we need to parse/route each packet.

- **STUN**: We use can choose a unique `ufrag` and route packets based on it.
- **SRTP/SRTCP**: The browser chooses a random `ssrc` (u32)... we can *usually* route based on that.
- **DTLS**: Uh oh. We pray that [RFC9146](https://www.rfc-editor.org/rfc/rfc9146.html) gets widespread support.
- **TURN**: IDK I've never implemented it.

**Fun fact**: Browsers can randomly generate the same `ssrc`; it's only 4 bytes long.
If there is a collision, and no source IP/port mapping is available, Discord attempts to decrypt the packet with each possible decryption key.
If the key worked, hey we identified the connection!

But OpenAI doesn't even look at RTP/RTCP:
> No protocol termination: Relay parses only STUN headers/ufrag; it uses cached state for subsequent DTLS, RTP, and RTCP, keeping packets opaque.

AKA "we really hope the source IP/port never changes".

Thats not a problem per-se, source address migration is relatively rare and it's the status quo for TCP.
But it feels gross to have this much complexity just to *downgrade* WebRTC.

## QUIC Fixes This
Remember that link to [RFC9146](https://www.rfc-editor.org/rfc/rfc9146.html)?
In the DTLS section?
Good times.
It's literally copied from QUIC.

If you're tight in the networking nerd circle, QUIC famously ditches source IP/port based routing.
Every packet instead contains a `CONNECTION_ID`, which can be 0-20 bytes long.
And most importantly: **it's chosen by the receiver**.

So our QUIC server generates a unique `CONNECTION_ID` for each connection.
Cool, we can use a single port and support connection migration, problem solved.
None of this complicated WebRTC protocol demuxing and custom routing bullshit.

But if your gut reaction is: "how dare they! this is a waste of bytes!"
These bytes are *very* important, keep reading u nerd.

## Stateless Load Balancing
I glossed over this, but OpenAI's load balancers (like most) depend on *shared state*.
Even if you have a sticky packet router, load balancers can still restart/crash and need to failover.
They're using a Redis instance to store the mapping of source IP/port to backend server, simple and easy.

But do you know what is even simpler and easier?
[QUIC-LB](https://www.ietf.org/archive/id/draft-ietf-quic-load-balancers-21.html) *uses the `CONNECTION_ID`* to store the backend server:

- The client initiates a QUIC connection.
- A load balancer receives the QUIC INITIAL packet, chooses a backend server, and encodes that decision into a unique `CONNECTION_ID`.
- The client echoes the `CONNECTION_ID` on every subsequent packet.
- A load balancer decodes the `CONNECTION_ID` and routes the packet to the appropriate backend server.

It works!
Just remember to encrypt the ID if you don't want to allow users to target arbitrary servers.

*Zero state* also means *zero global state*.
Theese load balancers could listen on a global anycast address, forwarding packets from anywhere in the world to the indicated backend server.
Cloudflare uses this extensively; no need for a global Redis cluster.

**Unpaid Shill**: [AWS NLB](https://aws.amazon.com/about-aws/whats-new/2025/11/aws-network-load-balancer-quic-passthrough-mode/) offers QUIC load balancing using QUIC-LB.
Other cloud providers need to step up their game and offer it too.

## Unicast in 2026
Based on the OpenAI blog, it sounds like they assign connections to regional load balancers.

Lame.
[Anycast](https://en.wikipedia.org/wiki/Anycast) is way cooler.

I brought this up in my ancient [Quic Powers](https://moq.dev/blog/quic-powers/) blog post, but I'll excuse you for not reading it (yet).
QUIC has something called `preferred_address` that is a game changer for load balancing.

Let's say we have thousands of servers around the world that could handle a new connection.
We have them all advertise the same anycast address, ex. `1.2.3.4`.
When a client tries to connect to `1.2.3.4`, the magic internet routers forward the packet to one of the servers.

Now, we could just use QUIC-LB and route traffic to the indicated backend.
But that would be boring.

Instead, we can give each QUIC server a unique unicast address, ex. `5.6.7.8`.
The idea is that we use anycast for handshakes and unicast for stateful connections.

- Server: Listen for QUIC packets on `1.2.3.4` and `5.6.7.8`.
- Client: Sends a QUIC handshake packet to `1.2.3.4`.
- Server: Establishs the QUIC connection, indicating `preferred_address=5.6.7.8`.
- Client: Sends future packets to `5.6.7.8`.

When the server is overloaded and doesn't want more connections, it stops advertising `1.2.3.4`.
We won't drop existing connections because they're safe on unicast.

Just like that, no load balancers needed!
The anycast address is basically a health check!

## Round Trips and U
The OpenAI blog post starts with a requirement:
> Fast connection setup so a user can start speaking as soon as a session begins

lol

Assuming a cold start, it takes 8\* round trips to establish a WebRTC connection.

Signaling server (ex. [WHIP](https://www.rfc-editor.org/rfc/rfc9725.html)):
- 0-1 for DNS (cached)
- 1 for TCP
- 1 for TLS 1.3
- 1 for HTTP

Media server:
- 1 for STUN (with server)
- 1 for ICE
- 2 for DTLS 1.2
- 1 for SCTP

\* It's complicated to compute, because some protocols can be pipelined to avoid 0.5 RTT.
Kinda like the [A-Button Challenge](https://www.youtube.com/watch?v=kpk2tdsPh0A).

All of this nonsense is because WebRTC needs to support P2P.
It doesn't matter if you have a server with a static IP address, you still need to do this dance.
It's extra depressing when the signaling and media server are running on the same host/process; you end up doing two (expensive) TLS handshakes.

QUIC fixes this of course:
- 0 for DNS (cached)
- 1 for QUIC
- 1 for HTTP/3 or WebTransport

\* Technically, QUIC and TLS 1.3 can support 0-RTT for resumption, but there are security caveats.

If you really need P2P, layer something like [Iroh](https://www.iroh.computer/) on top of QUIC.

## What's The Point?
Okay we're way too deep in the weeds, let's zoom out a bit.
Even if we ignore these technical barbs:

**WebRTC is a poor fit for Voice AI.**

Huh?
But it seems like a natural fit?
WebRTC is for conferencing, and that involves speaking, right?

...but there are core problems.

## TTS is faster than real-time
You speak into the microphone, it gets sent to one of OpenAI's billion servers, and they reply via text-to-speech.

Let's say it takes 1s of compute to generate 8s of audio.
In an ideal world, we would stream the audio as it's being generated (1s elapsed) and the client will slowly play it back (8s elapsed).
That way, if there's a network blip, well I already have audio buffered locally so it doesn't matter.

But nope, WebRTC *has no buffering* and renders based on arrival time.
OpenAI has to pace transmissions so it takes 8s to deliver the audio.
This means adding a sleep in front of every audio packet *before sending it*.

Believe it or not, this **actually increases latency**.
WebRTC has a dynamic (audio) jitter buffer that can be sized anywhere from 20ms to 200ms, based on the RTT.
The idea is to smooth out network variance... but we're the ones causing the problem!

If you instead use WebSocket/WebTransport to stream the audio faster than real-time, you can avoid this jitter buffer.

## WebRTC is too aggressive
WebRTC reduces latency by aggressively dropping packets.
If you've ever heard distorted audio on a conference call, that's WebRTC.

But I'd argue there's a product+techoloy mismatch here.
All of this expensive inference is being wasted when WebRTC decides to drop a packet.

Let's say I'm speaking to ~~Scarlett Johansson~~ Sky and I ask "how's your day going honey?".
If my input is garbled, I don't want to waste my tokens on equally garbled output.
Seriously, I'd rather wait an extra 100ms to retransmit these lost packets.

But guess what?
It's *impossible* to retransmit lost audio packets with WebRTC.
We tried at Discord, none of our hacks worked.
The protocol is designed to transmit an audio packet a single time.

And like I said above, it's just dumb that TTS replies can be dropped at all.
You're intentionally hurting the user experience by dropping packets to keep latency low... but the server artifically introduced that latency.

Like I just don't understand why you would ever use WebRTC for voice AI
Yes, Voice AI agents will eventually get the latency down to the conversational range, but we're not there yet
The inference still takes 500ms+, it's not worth dropping audio packets.

## But What Instead?
If not WebRTC, then what should you use for Voice AI?

Honestly, just stream audio over WebSockets.
You can leverage existing TCP/HTTP infrastructure instead of inventing a custom WebRTC load balancer.
You'll have higher quality and potentially even lower latency.
It's simple and works for OpenAI scale.

The day might come when you want to drop audio packets.
Or you want to transmit video over the same connection.
Then you can switch to WebTransport or MoQ.
